# S3 / MinIO

Proxera can expose S3-compatible services such as MinIO when clients use path-style bucket addressing.

This guide assumes a MinIO service reachable by the agent at a LAN address such as `http://192.168.1.248:80`, exposed publicly as `https://s3.intranet.example.com`.

---

## Route Settings

Create a route with:

| Field | Example |
|-------|---------|
| Domain | `s3.intranet.example.com` |
| LAN Host | `192.168.1.248` |
| Port | `80` |
| Path prefix | `/` or blank |
| Strip prefix | disabled |
| Preserve public Host header | enabled |

The **Preserve public Host header** option is required for S3 signature validation. S3 SigV4 includes the public `Host` value in the canonical request, so the agent must connect to the LAN target while preserving the client-facing host.

Use path-style addressing:

```text
https://s3.intranet.example.com/my-bucket/object-name
```

Bucket subdomains such as `https://my-bucket.s3.intranet.example.com/object-name` require wildcard DNS, wildcard ingress routing, and matching Proxera routes. They are not the recommended setup.

---

## Ingress Requirements

Large S3 uploads must pass through the Kubernetes ingress before they reach Proxera. With nginx ingress, the default request body limit is too low for backup workloads and returns:

```text
413 Request Entity Too Large
```

Add these annotations to the public S3 ingress:

```yaml
nginx.ingress.kubernetes.io/proxy-body-size: "512m"
nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"
```

For example:

```bash
kubectl -n proxera annotate ingress minio \
  nginx.ingress.kubernetes.io/proxy-body-size="512m" \
  nginx.ingress.kubernetes.io/proxy-read-timeout="3600" \
  nginx.ingress.kubernetes.io/proxy-send-timeout="3600" \
  --overwrite
```

Choose a `proxy-body-size` larger than the largest S3 upload part your client sends. For StackGres backups, this must be large enough for each backup tar partition or S3 multipart part.

## Client Configuration

Configure S3 clients for:

| Setting | Value |
|---------|-------|
| Endpoint | `https://s3.intranet.example.com` |
| Addressing style | path-style |
| Region | the MinIO/S3 region, commonly `us-east-1` |
| TLS | enabled at the public ingress |

For StackGres S3-compatible storage, the relevant fields are:

```yaml
storage:
  type: s3Compatible
  s3Compatible:
    bucket: backups
    endpoint: https://s3.intranet.example.com
    region: us-east-1
    enablePathStyleAddressing: true
```

## Known Limits

Proxera's current HTTP tunnel buffers request bodies before forwarding them to the agent. This works for normal uploads, but very large S3 backup parts can put pressure on the Proxera pod heap because the payload is also encoded for the tunnel.

For production backup workloads:

- Set the ingress body limit explicitly.
- Keep S3 part sizes below the Proxera pod memory budget.
- Prefer multipart configurations with bounded part size.
- Monitor Proxera pod memory during the first full backup.

Streaming large uploads through the tunnel without buffering is future work.

## Troubleshooting

**Small uploads work, large uploads fail with HTML 413**

The request is being rejected by nginx ingress before Proxera or MinIO can handle it. Add or increase `nginx.ingress.kubernetes.io/proxy-body-size` on the S3 ingress.

**Requests fail with `SignatureDoesNotMatch`**

Enable **Preserve public Host header** on the Proxera route and verify the client signs requests for the same public endpoint it is calling.

**Bucket listing works, but object operations fail**

Check the MinIO policy for the access key. A valid signature can still return `AccessDenied` when the key lacks `s3:GetObject`, `s3:PutObject`, or `s3:DeleteObject`.

**StackGres backup fails while uploading a part**

Inspect the backup pod logs. If the error body contains `<html>` and `413 Request Entity Too Large`, the ingress body-size annotation is missing or too small.
